Question #802
A company wants to run its payment application on AWS. The application receives payment notifications from mobile devices.
Payment notifications require a basic validation before they are sent for further processing.
The backend processing application is long running and requires compute and memory to be adjusted. The company does not
want to manage the infrastructure.
Which solution will meet these requirements with the LEAST operational overhead?

进一步处理。
后端处理应用程序是长时间运行的，并且需要调整计算和内存。公司不希望管理基础设施。

题目分析与解答
1. 考察的知识点
无服务器架构（Serverless Architecture）
AWS服务集成：Amazon API Gateway、AWS Lambda、Amazon ECS、AWS Fargate
最小化运维开销（Least Operational Overhead）
事件驱动架构（Event-Driven Architecture）
2. 选项分析
A:
使用Amazon SQS和Amazon EventBridge进行消息处理是合理的，但后端应用程序部署在Amazon EKS Anywhere需要用户管理基础
设施（独立集群），这增加了运维开销。
错误原因：Amazon EKS Anywhere并非完全托管服务，违反了“最小化运维开销”的要求。
B:
使用Amazon API Gateway和AWS Step Functions进行消息处理是合理的，但后端应用程序部署在Amazon EKS（自管理节点）需要
用户管理节点和集群，增加了运维开销。
错误原因：EKS自管理节点需要用户负责节点的维护，违反了“最小化运维开销”的要求。
C:
使用Amazon SQS和Amazon EventBridge进行消息处理是合理的，但后端应用程序部署在Amazon EC2 Spot Instances需要用户管理
实例和Spot Fleet，增加了运维开销。
错误原因：EC2 Spot Instances需要用户负责实例管理，违反了“最小化运维开销”的要求。
D:
使用Amazon API Gateway和AWS Lambda进行消息处理是合理的，后端应用程序部署在Amazon ECS（使用AWS Fargate启动类型）
是完全托管的解决方案，符合“最小化运维开销”的要求。
正确原因：AWS Fargate是无服务器计算引擎，用户无需管理底层基础设施，完全符合题目要求。
