题目解析：多层架构下位置数据的存储与检索（REST API 可访问）
1. 本题考察的知识点
API 层：如何让数据点可通过 REST API 访问（典型为 Amazon API Gateway）。
计算/业务逻辑层：如何在无服务器或多层架构中实现“写入位置数据 + 查询位置数据”的业务逻辑（典型为 AWS Lambda）。
存储与低延迟访问：位置数据属于高峰期高并发写入、需要被 API 低延迟读取的典型场景（常见落地是 DynamoDB/Redis/S3 等，
但本题选项未给出存储服务，需判断哪个组合最贴近“通过 REST API 存取数据”的架构模式）。
分析平台对接：数据后续要进入既有 analytics 平台，但本题问的是“用于存储与检索位置数据”的动作/方案，而不是 BI 可视化或离
线查询。

API 要提供 REST 访问，因此前门最合理是 API Gateway。其次需要一个后端来执行业务逻辑：接收位置点、写入数据存储，并在
查询时从存储读取并返回。选项里只有 API Gateway + Lambda 具备“REST API + 业务逻辑处理”这一完整形态。 虽然“存储”服务没
有出现在选项中，但在 AWS 标准模式里，Lambda 通常作为控制器/服务层，对接 DynamoDB/S3/RDS 等实现真正存取。因此在给
定选项中，最可行的是 B。 其他选项要么不是 REST 的存取模式，要么是分析/可视化组件而非在线存取组件。
3. 逐个分析每个选项为什么正确或错误
A. Use Amazon Athena with Amazon S3 错误。 原因：
Athena 是对 S3 上的数据做交互式 SQL 查询（偏离线/交互式分析），不是为 REST API 提供低延迟在线查询而设计。
S3 + Athena 不适合高峰期大量位置点的“实时写入 + API 频繁读取”的在线交易型访问模式（延迟、数据组织方式、查询启动开销等
都不匹配）。
题目强调“数据点必须可从 REST API 访问”，该组合不直接提供 REST API 接入与在线读写路径。
B. Use Amazon API Gateway with AWS Lambda 正确（在给定选项中最符合）。 原因：
API Gateway 直接用于构建与发布 REST API，满足“数据点必须可从 REST API 访问”。
Lambda 承载业务逻辑：接收位置上报请求、校验/转换数据、并把数据写入后端存储；在查询时从后端存储读取并返回给 API。
可天然构建“多层架构”（API 层 + 计算层 + 数据层）。尽管选项未写出数据层，但 Lambda 作为中间层是标准做法。
具备弹性扩缩，适配“peak operating hours”的突发高并发（前提是后端存储也选用可扩展方案）。
C. Use Amazon QuickSight with Amazon Redshift 错误。 原因：
QuickSight 是 BI 可视化，不用于提供 REST API 的在线读写接口。
Redshift 是数据仓库，适合分析型查询，但并非此题强调的“REST API 可访问的在线存取位置数据点”的第一选择。
该组合更像“数据进仓 + 报表分析”，不能直接解决 REST API 存取位置点的需求。
D. Use Amazon API Gateway with Amazon Kinesis Data Analytics 错误。 原因：
Kinesis Data Analytics 主要用于对流数据做实时分析（SQL/Flink），它不是典型的“存储 + API 查询”后端。
即使可以通过 API Gateway 接入，也缺少“持久化存储与按需查询”的核心闭环；Kinesis Data Analytics 本身不作为数据存储系统。
若要实现该场景，通常还需要 Kinesis Data Streams/Firehose + DynamoDB/S3/OpenSearch 等组件，选项未体现。
