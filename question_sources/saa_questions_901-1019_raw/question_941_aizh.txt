题目分析与解答
1. 考察的知识点
本题考察的是 Amazon Elastic Kubernetes Service (Amazon EKS) 中 IAM Roles for Service Accounts (IRSA) 的使用。
IRSA 是一种机制，允许 Kubernetes 服务账户通过 IAM 角色安全地访问 AWS 资源。
重点知识包括 IAM 角色的定义、权限的分配以及 OpenID Connect (OIDC) 身份提供商的信任关系设置。
2. 选项分析
A. Create an IAM policy that defines the required permissions Attach the policy directly to the IAM role of the EKS nodes.
错误：将 IAM policy 附加到 EKS 节点的 IAM role 是一种粗粒度的权限分配方式，无法实现 Kubernetes 服务账户的细粒度权限控
制。
IRSA 的目标是为每个服务账户分配独立的 IAM role，而不是依赖节点角色。
B. Implement network policies within the EKS cluster to prevent Kubernetes service accounts from accessing specific AWS
services.
错误：网络策略主要用于控制 Pod 之间的网络流量，而不是管理服务账户对 AWS 资源的访问权限。
IRSA 是通过 IAM 角色和 OIDC 身份提供商来实现权限控制，与网络策略无关。
C. Modify the EKS cluster's IAM role to include permissions for each Kubernetes service account. Ensure a one-to-one
mapping between IAM roles and Kubernetes roles.
错误：EKS 集群的 IAM role 是为集群本身设计的，不能直接用于服务账户的权限分配。
IRSA 的设计是通过独立的 IAM role 和服务账户的映射来实现权限控制，而不是修改集群角色。
D. Define an IAM role that includes the necessary permissions. Annotate the Kubernetes service accounts with the
Amazon Resource Name (ARN) of the IAM role.
正确：这是 IRSA 的核心步骤之一。通过定义 IAM role 并将其 ARN 注解到 Kubernetes 服务账户，可以实现服务账户与 IAM role 的
绑定。
这种方式允许服务账户以该 IAM role 的身份访问 AWS 资源。
E. Set up a trust relationship between the IAM roles for the service accounts and an OpenID Connect (OIDC) identity
provider.
正确：这是 IRSA 的另一个关键步骤。通过设置 IAM role 与 OIDC 身份提供商之间的信任关系，AWS 可以验证服务账户的身份并授
予相应权限。
OIDC 身份提供商通常由 EKS 集群自动配置，用于支持 IRSA。
