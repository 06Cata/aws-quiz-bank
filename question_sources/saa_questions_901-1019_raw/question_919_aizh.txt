Question #919
A company recently performed a lift and shift migration of its on-premises Oracle database workload to run on an Amazon EC2
memory optimized Linux instance. The EC2 Linux instance uses a 1 TB Provisioned IOPS SSD (io1) EBS volume with 64,000 IOPS.
The database storage performance after the migration is slower than the performance of the on-premises database.
Which solution will improve storage performance?
一家公司最近将其本地的 Oracle 数据库工作负载迁移到 Amazon EC2 内存优化的 Linux 实例上运行。该 EC2 Linux 实例使用了一个
1 TB Provisioned IOPS SSD (io1) EBS 卷，具有 64,000 IOPS。
迁移后的数据库存储性能比本地数据库的性能慢。
哪种解决方案可以提高存储性能？
A. Add more Provisioned IOPS SSD (io1) EBS volumes. Use OS commands to create a Logical Volume Management (LVM)

题目分析与解答
1. 考察的知识点
Amazon EC2实例与EBS卷的性能优化。
Provisioned IOPS SSD (io1) EBS卷的性能特性。
使用逻辑卷管理 (LVM) 实现存储条带化以提高性能。
2. 选项分析
A. Add more Provisioned IOPS SSD (io1) EBS volumes. Use OS commands to create a Logical Volume Management (LVM)
stripe.
正确：通过添加更多的io1卷并使用LVM条带化，可以实现并行读写操作，从而显著提高存储性能。
条带化是一种常见的性能优化技术，尤其适用于高性能数据库工作负载。
B. Increase the Provisioned IOPS SSD (io1) EBS volume to more than 64,000 IOPS.
错误：单个io1卷的最大IOPS限制为64,000（在Nitro系统上为256,000），因此无法通过增加IOPS超过64,000来解决问题。
即使增加IOPS到最大值，也可能无法满足数据库的性能需求。
C. Increase the size of the Provisioned IOPS SSD (io1) EBS volume to 2 TB.
错误：增加卷的大小不会直接提高IOPS性能。IOPS与卷大小相关，但问题的核心是性能不足，而不是存储容量不足。
D. Change the EC2 Linux instance to a storage optimized instance type. Do not change the Provisioned IOPS SSD (io1) EBS
volume.
错误：存储优化实例类型（如I3或I3en）通常使用本地NVMe存储，而不是EBS卷。如果继续使用EBS卷，改变实例类型不会显著提
高性能。
问题的核心在于EBS卷的性能，而不是实例类型。
