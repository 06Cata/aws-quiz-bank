题目分析与解答
1. 考察的知识点
Amazon EKS 与 AWS Fargate 的存储解决方案。
数据持久性、高可用性和容器间共享存储的实现。
对比 Amazon EBS 和 Amazon EFS 的特性及适用场景。
操作开销的最小化。
2. 选项分析
A. Create Amazon Elastic Block Store (Amazon EBS) volumes in the same Availability Zones where EKS worker nodes are
placed. Register the volumes in a StorageClass object on an EKS cluster. Use EBS Multi-Attach to share the data between
containers.
Amazon EBS 是一种块存储服务，主要用于单个实例的存储需求。
EBS Multi-Attach 允许多个 EC2 实例挂载同一个 EBS 卷，但不支持 Fargate。
此外，EBS 的可用性受限于单个可用区，无法满足高可用性和容器间共享存储的需求。
错误原因：不支持 Fargate，且无法跨可用区实现高可用性。
B. Create an Amazon Elastic File System (Amazon EFS) file system. Register the file system in a StorageClass object on an
EKS cluster. Use the same file system for all containers.
Amazon EFS 是一种文件存储服务，支持跨多个可用区的高可用性和容器间共享存储。
EFS 可以通过 StorageClass 对象轻松集成到 EKS 集群中，且支持 Fargate。
操作开销较低，因为 EFS 是托管服务，用户无需管理底层基础设施。
正确原因：满足高可用性、容器间共享存储需求，并且操作开销最低。
C. Create an Amazon Elastic Block Store (Amazon EBS) volume. Register the volume in a StorageClass object on an EKS
cluster. Use the same volume for all containers.
与选项 A 类似，EBS 是块存储服务，主要用于单实例存储需求。
单个 EBS 卷无法跨多个容器共享，且不支持 Fargate。
此外，EBS 的可用性受限于单个可用区，无法满足高可用性需求。
错误原因：不支持 Fargate，且无法跨容器共享存储。
D. Create Amazon Elastic File System (Amazon EFS) file systems in the same Availability Zones where EKS worker nodes
are placed. Register the file systems in a StorageClass object on an EKS cluster. Create an AWS Lambda function to
synchronize the data between file systems.
创建多个 EFS 文件系统并使用 Lambda 同步数据会增加操作开销。
EFS 本身已经支持跨可用区的高可用性，无需额外的同步机制。
此选项复杂度较高，不符合“最小操作开销”的要求。
错误原因：增加了不必要的复杂性和操作开销。
